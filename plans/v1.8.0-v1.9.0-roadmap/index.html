    v1.8.0–v1.9.0 Roadmap¶

    Deux features incrémentales, propres, indépendantes, construites sur la base actuelle de main (v1.7.5).

    v1.8.0 — JSON Handshake¶

    Remplacer la poignée de main HELLO/HELLO_ACK (via MessageBuffer) par un échange JSON brut sur le socket TCP nu, AVANT le protocole binaire.

    Architecture¶

    TCP connect()
  ↓
Phase 1 — JSON handshake (raw socket, [2B length][JSON])
  ↓  do_server_handshake() / do_client_handshake()
Phase 2 — Binary protocol (MessageBuffer, MAGIC+header+content)

    Wire format du handshake (immutable à vie) : 

    [2 bytes payload length (big-endian)]
[N bytes UTF-8 JSON payload]

    Payload serveur → client : 

    {"v": "1.8.0", "meta": {}}

    Payload client → serveur : 

    {"v": "1.8.0", "meta": {}}

    v1.9.0 — ID System¶

    Ajout d'un système d'allocation d'IDs par fenêtres (IdManager + IdResolver), remplacement du request_id aléatoire par des IDs séquentiels.

    Architecture¶

    IdManager (thread-safe, range allocator)
  └── allocate() → base_id
        └── IdResolver (per-client)
              ├── client_range = [0, half)
              │     └── client émet : request_id = base + offset
              └── server_range = [half, window_size)
                    └── serveur émet : request_id = base + (offset - half)

    Nouveau fichier¶

    internal/id_manager.py¶

    class IdManager:
    """Thread-safe range allocator."""
    __slots__ = ("_next_base", "window_size", "_lock")
    def __init__(self, window_size: int = 512) ...
    def allocate(self) -> int: ...

class IdResolver:
    """Per-client resolver, splits window in half."""
    __slots__ = ("id_manager", "base", "half", ...)
    def __init__(self, id_manager: IdManager) ...
    def get_id(self, mode: Mode) -> int: ...
    def resolve_client_request(self, id_: int) -> int: ...
    def resolve_server_request(self, id_: int) -> int: ...

    Ce qui NE change PAS¶

    • Le format du handshake JSON (Phase 1) — toujours [2B length][JSON]
    • Les callbacks utilisateur : ils reçoivent toujours Response avec request_id
    • L'API publique : send_and_wait(), ping_client(), etc. inchangées

    Dépendances entre les deux versions¶

    Dépendance v1.8.0 → v1.9.0
    Handshake JSON Requis pour v1.9.0 (la meta transmet window_size/base_id)
    IdManager Indépendant (peut être ajouté après)
    Changement header struct Indépendant (v1.9.0 seulement)
    Résolution des IDs Dépend du handshake JSON + IdManager

    Ordre : v1.8.0 d'abord, puis v1.9.0.

    Notes importantes¶

    1. Breaking changes : Les deux versions sont incompatibles avec les versions précédentes (v1.7.x). Les clients et serveurs doivent être mis à jour ensemble.
    2. Pas de rétrocompatibilité entre v1.8.0 et v1.9.0 : le header struct change (14B vs 16B). Les composants v1.8.0 ne peuvent pas parler à des v1.9.0.
    3. Code réutilisable : La branche v1.8.0-idmanager (supprimée) contenait ~70% du code final. L'implémentation doit repartir de zéro sur main en suivant ce plan.
    4. Tests : Chaque version doit être testée intégralement avant de passer à la suivante.
